Three personas — the shop's two, and the operator above them

The medium-enterprise line splits work across five roles. A Lao SME is run by one person who is owner, manager, HR and bookkeeper at once — so inside the shop this branch collapses to two: a focused Staff Portal and a single Owner console. Above every shop sits a third persona — the Platform Administrator (you, the operator) — who approves registrations, watches each tenant's resources, and allocates capacity. Two personas serve the tenant; the third runs the platform.

§2

Multi-tenant on the free tier — shared database, tenant-scoped rows

Many companies, one deployment, near-zero marginal cost. Every operational row carries a tenant_id; a new tenant registry holds each company's profile, plan and quotas; the edge Worker stamps and enforces the tenant on every query. Sensitive stores stay custody-sealed exactly as in the parent.

Topology

  • Static — GitHub repo → Pages (the SPA, free)
  • API — one Cloudflare Worker, free tier, deploy-on-push
  • Data — one Turso DB, shared schema, all tenants
  • Sealed — identity server-authoritative (edge.auth custody flip)

Shared-schema is the cheapest topology: it packs the most companies into a single free database, where DB-per-tenant would exhaust the free DB count quickly.

The tenant stamp

Sign-in resolves account → tenant_id. The Worker injects it into the session and into every read/write; a row-level guard rejects any query missing or crossing a tenant. The browser never chooses its own tenant_id.

session = { uid, tenant_id, scopes }
WHERE tenant_id = :session.tenant_id — appended by the data layer, not the caller

Isolation & the trade-off

  • Row-level scoping on all ops stores
  • Per-tenant backup, export & hard delete (purge by tenant_id)
  • Sensitive stores (identity · payroll · ledger · tax) sealed + tenant-scoped

watch Shared DB = shared blast radius & noisy-neighbour risk. Mitigation: query guard tested on every path, per-tenant rate & quota caps (§7/§2.4), and an upgrade path to isolated DB for a paying tenant that needs it.

Tenant onboarding — KYC register, then manual activation

Not instant self-serve: an owner proves who they are, and a Platform Administrator (§4P) activates the tenant by hand. This keeps fake/abusive tenants off the shared free database — the gate that lets "free" stay safe.

Register

Company name, email, phone, language — plus an ID-card photo and an owner selfie/screen-capture to match the ID.

Pending

Tenant created status=pending; KYC images sealed in db_registration. Nothing operational yet.

Platform review

Platform Admin compares selfie ↔ ID, confirms email/phone → Activate or Reject (reason logged).

Provision

On activate: status=active, owner account (server-auth), free plan + seats 10/3/1, Lao tax/leave defaults seeded.

Live

Owner signs in, adds managers & staff; clock-in works offline day one.

KYC data is sensitive. ID-card and selfie images live in a sealed, tenant-scoped store (db_registration), encrypted at rest, access logged as export.*/kyc.* facts, with a retention rule. Only the Platform Admin reviewing the application can view them — and that view is audited.

Carried from research — the punch is sacred

Never-block

A punch never fails on patchy 3G — it queues locally and syncs; the server stamps the authoritative time on arrival.

Photo, not biometric

Selfie stored as a human-reviewed photo by default (no face template) — stays below the ASEAN biometric-consent bar.

Fair by design

A GPS/selfie flag routes to the owner's review queue — it never auto-docks pay. Anti-fraud as deterrent, not punishment.

Optional .sysmatik add-on fuses terminal (ZKTeco/Hikvision) + kiosk + this mobile punch into one ledger. See the Live Check-in brief and v2.4.1.sysmatik.2.

Geofenced clock-in — the manager draws the circle

§5

Payroll — split into a Lao-correct pipeline

The heart of this branch. Most SME tools get payroll "roughly right" and leave the owner exposed at NSSF and Tax Department filing time. Adeptio computes it as a transparent, auditable pipeline where every line is shown, the statutory math follows current Lao law, and the owner picks how deep to go.

The pipeline — five stages, two sides

① Inputs

Comp profile (base + allowances), attendance & OT, leave (unpaid), one-offs (bonus · advance · fine).

② Gross

base + allowances + OT (Lao multipliers) + bonus = gross earnings.

③ Social security

NSSF base = min(earnings, cap). Employee 5.5% deducted; employer 6% added to cost.

④ PIT

Taxable = gross − employee NSSF. Progressive 0–25% by monthly bracket.

⑤ Net & cost

Net = gross − NSSF − PIT − other. Employer cost = gross + employer NSSF.

Two numbers, never confused: what the worker takes home (net) and what the company spends (employer cost = gross + 6% NSSF). The second is what flows into Accounting as staff cost (§6) — the source of the cost/benefit linkage you asked for.

Accounting — the books live inside the system

No more keeping pay in one place and money in a spreadsheet. The owner records revenue and expenses by day, week or month right here; tax is computed from the same data; and because payroll already posts staff cost, the monthly cost/benefit picture builds itself. The spreadsheet becomes an export, never the source of truth.

Cashbook

One simple ledger of money in and out. Each entry:

  • Date · type (revenue / expense)
  • Category · amount (₭) · note
  • Tax flag (VAT in/out · exempt)
  • Method (cash · transfer · QR)
  • Optional receipt photo

Views: day (cash drawer), week (trend), month (close).

Auto-fed lines

Some lines the owner never types — they flow from elsewhere:

  • Salaries ← each closed pay run (§5)
  • Employer NSSF ← pay run (6%)
  • PIT / NSSF payable ← statutory
  • Channel fees ← LINE/WhatsApp metering (§7)

This wiring is what makes cost/benefit real instead of a guess.

Free tier & metering — how the money works

Free forever

The whole platform — personas, attendance, payroll, accounting, in-app + email — runs on free infrastructure (Workers · Turso · Pages). A small shop pays nothing to operate.

Metered add-ons

LINE & WhatsApp messages carry a per-message provider cost, so each tenant gets a monthly allowance, then tops up or upgrades. The meter lives in db_license; usage shows in §2.4.

Honest billing

Each send debits the tenant's quota and posts a fee line to Accounting (§6). The owner always sees what a channel costs before the bill — no surprise overage.

Metering path: send(template, channel) → check db_license.quota[channel] → debit 1 → log db_comms → post fee db_ledger → if quota low: alert owner
Open decision (DS7). The free-seat threshold and the LINE/WhatsApp per-message price (pass-through vs. small margin) are commercial calls, not technical ones — flagged in §11 for you to set. The plumbing meters whatever numbers you choose.

BO-4Payroll engine coregross · NSSF · PIT · payslip
stores / fields
db_people: comp_profile{ base, allowances[], pay_freq, ss_eligible }
db_payroll: run{ id, period, state }, slip{ uid, gross, ss_emp, ss_er,
            taxable, pit, net, lines[] }
calc (pure fn — table-driven from db_tax)
gross   = base + Σallowances + OT + bonus
ss_base = min(ss_eligible_earnings, CAP=4,500,000)
ss_emp  = ss_base * 0.055 ;  ss_er = ss_base * 0.06
taxable = gross - ss_emp
pit     = Σ bracket_slice(taxable) * rate   // 0/5/10/15/20/25
net     = gross - ss_emp - pit - other
cost    = gross + ss_er
test vector (must pass exactly)
base 6,000,000 → ss_emp 247,500 · ss_er 270,000 · taxable 5,752,500
              → pit 260,250 · net 5,492,250 · cost 6,270,000
reconcile: net + (ss_emp+ss_er) + pit == cost  ✓
